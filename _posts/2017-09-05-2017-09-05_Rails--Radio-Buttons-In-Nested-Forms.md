<h3 name="6875" id="6875" class="graf graf--h3 graf--leading graf--title">Rails: Radio Buttons In Nested Forms</h3>
<h4 name="435e" id="435e" class="graf graf--h4 graf-after--h3 graf--subtitle">The story behind my first Stack Overflow answer.</h4>
<p name="cd7a" id="cd7a" class="graf graf--p graf-after--h4">I was recently asked to add a feature to a Rails app. The app is a Contact Management System (CMS). Each contact has a name, an email address, and a phone number. I was asked to add the ability for multiple phone numbers to be added to each contact; each phone number should be either a home number, work number, or cell-phone number. Also, one of the numbers should be designated the default number.</p>
<p name="0703" id="0703" class="graf graf--p graf-after--p">The way I approached it was by moving the phone numbers into a separate <code class="markup--code markup--p-code">PhoneNumber</code> class, with a many-to-one relationship to the <code class="markup--code markup--p-code">Contact</code> class, so now every contact could have many phone numbers, and every phone number belongs to a contact (for more on one-to-many associations in Rails see my blog-post <a href="https://blog.yechiel.me/generating-belongs-to-associations-in-rails-be7b7fdea96c" data-href="https://blog.yechiel.me/generating-belongs-to-associations-in-rails-be7b7fdea96c" class="markup--anchor markup--p-anchor" rel="noopener" target="_blank">rails generate associations</a>).</p>
<p name="6b1b" id="6b1b" class="graf graf--p graf-after--p">I now had to update the ‘New Contact’ form to accept the nested attributes for the <code class="markup--code markup--p-code">PhoneNumber</code> class, as well as add a radio button so users can select which number would be the default, and this is where I ran into a problem.</p>
<h4 name="ba74" id="ba74" class="graf graf--h4 graf-after--p">No soap, radio!</h4>
<p name="f648" id="f648" class="graf graf--p graf-after--h4">First a brief explanation of how radio buttons work:</p>
<p name="3837" id="3837" class="graf graf--p graf-after--p">If you look at the HTML for a form with radio buttons, you will see that each button is it’s own input element, what links them together is that they all share the same <code class="markup--code markup--p-code">name</code> attribute. This is what it looks like:</p>
<figure name="7fce" id="7fce" class="graf graf--figure graf--iframe graf-after--p"><iframe src="https://codepen.io/yechielk/embed/preview/rzRRoZ?height=600&amp;slug-hash=rzRRoZ&amp;default-tabs=html,result&amp;host=https://codepen.io&amp;embed-version=2" width="700" height="525" frameborder="0" scrolling="no"></iframe></figure><p name="b89d" id="b89d" class="graf graf--p graf-after--figure">Notice how each <code class="markup--code markup--p-code">input</code> has the same name (<code class="markup--code markup--p-code">picker</code>), that’s what lets the browser know that they are all linked and that it should only allow one of the buttons to be picked at a time.</p>
<h4 name="a350" id="a350" class="graf graf--h4 graf-after--p">Rails Helpers</h4>
<p name="a8f1" id="a8f1" class="graf graf--p graf-after--h4">Rails has a built in helper method that generates radio buttons for forms; the <code class="markup--code markup--p-code">radio_button</code> method. When that method is called on a form element Rails will automatically generate the proper HTML element with the correct attributes so that when the form gets submitted your server will know how to construct a new object (or update an existing one) from the data.</p>
<p name="299b" id="299b" class="graf graf--p graf-after--p">My problem was that I wasn’t making a regular form, I was making a nested form, a form within a form. I had my Contact form, and within the Contact form, I was collecting attributes for my <code class="markup--code markup--p-code">PhoneNumber</code> class.</p>
<p name="94b1" id="94b1" class="graf graf--p graf-after--p">Rails generally handles that kind of stuff pretty well. What it does is, it adds fields to your form, and each field gets a name attribute that follows the format of <code class="markup--code markup--p-code">parent_class[nested_classes_attributes][:index][attribute]</code>, so in my case, where I had <code class="markup--code markup--p-code">PhoneNumber</code> nested in the <code class="markup--code markup--p-code">Contact</code> parent class, the Number field in the first phone number form had a name attribute of <code class="markup--code markup--p-code">contact[phone_numbers_attributes][0][number]</code>, the number field in the second phone number form had a `name` attribute of <code class="markup--code markup--p-code">contact[phone_numbers_attributes][1][number]</code>, and so on. Each form incrementing the index by one, so that the server would know we are dealing with different <code class="markup--code markup--p-code">PhoneNumber</code> objects.</p>
<p name="cd6e" id="cd6e" class="graf graf--p graf-after--p">The problem was that I needed a radio button on each phone number form so that users could choose which phone number was the default, but I needed all of the radio buttons to have the same name. The Rails form helper method, as mentioned, was treating each phone number as a separate form, and giving each button a different name attribute.</p>
<p name="3920" id="3920" class="graf graf--p graf-after--p">As usual, I turned to Google and was pretty excited to see that one of the first results linked to a question on Stack Overflow that sounded just like the problem I was having. The problem was that when I clicked on it, I saw that even though the question was first asked in 2011, it only had three answers, and none of them solved my problem.</p>
<figure name="9fc7" id="9fc7" class="graf graf--figure graf-after--p"><img class="graf-image" data-image-id="1*J3Ygm52dcu75LQkQ6McuEw.png" data-width="485" data-height="270" data-is-featured="true" src="/assets/images/posts/2017-09-05-2017-09-05_Rails--Radio-Buttons-In-Nested-Forms-0.png"><figcaption class="imageCaption">Image Credit: <a href="https://xkcd.com/979/" data-href="https://xkcd.com/979/" class="markup--anchor markup--figure-anchor" rel="noopener" target="_blank">XKCD</a></figcaption></figure><h4 name="ad79" id="ad79" class="graf graf--h4 graf-after--figure">Great Help!</h4>
<p name="5c7a" id="5c7a" class="graf graf--p graf-after--h4">An incomplete solution provided by one of the answers was that, instead of attaching the radio button to the existing form, to have it as an independent <code class="markup--code markup--p-code">radio_button_tag</code> and to manually supply the name as well as any other attributes needed. That would result in a form that looked like this:</p>
<pre name="2ac4" id="2ac4" class="graf graf--pre graf-after--p">&lt;%= form_for <a href="http://twitter.com/contact" data-href="http://twitter.com/contact" class="markup--anchor markup--pre-anchor" title="Twitter profile for @contact" rel="noopener" target="_blank">@contact</a> do |f| %&gt;<br> &lt;%= f.text_field :name %&gt;<br> &lt;%= f.email_field :email %&gt;<br> &lt;%= f.fields_for :phoneNumbers do |phone_form| %&gt;<br> &lt;%= phone_form.text_field :number %&gt;<br> &lt;%= phone_form.select :location, [“Home”, “Work”, “Mobile”, “Fax”] %&gt;<br> &lt;%= radio_button_tag “contact[default_telephone]” %&gt;<br> &lt;% end %&gt;<br>&lt;% end %&gt;</pre>
<p name="e28f" id="e28f" class="graf graf--p graf-after--pre">You will notice that the <code class="markup--code markup--p-code">:number</code> and <code class="markup--code markup--p-code">:location</code>fields are attached to a <code class="markup--code markup--p-code">phone_form</code> object which gives them their <code class="markup--code markup--p-code">name</code> attribute, but the radio button tag isn’t attached to anything, instead the name (<code class="markup--code markup--p-code">contact[default_telephone]</code>) is supplied as an argument.</p>
<p name="7ee4" id="7ee4" class="graf graf--p graf-after--p">The problem is that the radio buttons need two more pieces of information to work properly. The first piece of information was the value that would be submitted if the button was checked off, which would ideally be the index of the current phone number the form was rendering. The second piece of information was which phone number was the current default number so that the correct button would be selected by default if the form was an Edit Contact form. And those two pieces of information were missing from the Stack Overflow answer; I was on my own.</p>
<h4 name="9ebf" id="9ebf" class="graf graf--h4 graf-after--p">A Solution To Every Problem</h4>
<p name="dbb5" id="dbb5" class="graf graf--p graf-after--h4">I spent some time thinking about the problem and realized something. The other fields, the ones that were attached to <code class="markup--code markup--p-code">phone_form</code> were probably getting their information from the <code class="markup--code markup--p-code">phone_form</code> object. If my <code class="markup--code markup--p-code">radio_button_tag</code> was still within the <code class="markup--code markup--p-code">phone_form</code> scope (which it was, as long as I put it before the first <code class="markup--code markup--p-code">&lt;% end %&gt;</code> tag) it should have access to the <code class="markup--code markup--p-code">phone_form</code> object and any methods available to it.</p>
<p name="5e5f" id="5e5f" class="graf graf--p graf-after--p">I would have to find a way to inspect the <code class="markup--code markup--p-code">phone_form</code> object and see what was available through it.</p>
<p name="d8b9" id="d8b9" class="graf graf--p graf-after--p">One of my favorite debugging methods in rails is to put a <code class="markup--code markup--p-code">raise &lt;object&gt;.inspect</code> at a spot in my code, then when my app would reach that breakpoint it would display the object on the screen as an error, along with a console that would have access to any variables available at the breakpoint.</p>
<p name="0d30" id="0d30" class="graf graf--p graf-after--p">The problem was, I have only used that method in my controllers, I never used it in a view yet, and while I had a hunch it would work, I wanted to make sure. So to start with I put the following in the middle of my view:<br><code class="markup--code markup--p-code">&lt;% raise “Hello World”.inspect %&gt;</code> and navigated to that page. Sure enough, I got the following screen:</p>
<figure name="0115" id="0115" class="graf graf--figure graf-after--p"><img class="graf-image" data-image-id="1*VL-FQKFxudVzo1znUxdTJg.png" data-width="933" data-height="463" src="/assets/images/posts/2017-09-05-2017-09-05_Rails--Radio-Buttons-In-Nested-Forms-1.png"><figcaption class="imageCaption">Notice the “Hello World” near the top.</figcaption></figure><p name="9fd8" id="9fd8" class="graf graf--p graf-after--figure">Perfect, now let’s inspect the actual object. I put the following into my form, right before the radio button tag <code class="markup--code markup--p-code">&lt;% raise phone_form.inspect %&gt;</code>:</p>

<p name="96c9" id="96c9" class="graf graf--p graf-after--figure">Yeah, quite an eyeful. That’s what a Ruby Object looks like, and this object was apparently a <code class="markup--code markup--p-code">FormBuilder </code>object. All the information I needed was somewhere in the huge wall of text; the problem was how to find it.</p>
<p name="6ad6" id="6ad6" class="graf graf--p graf-after--p">I headed to the console at the bottom to see what I could find. Running <code class="markup--code markup--p-code">phone_form</code> just gave me the same wall of text, so I decided to run <code class="markup--code markup--p-code">phone_form.methods</code> to see what methods were available to the <code class="markup--code markup--p-code">phone_form</code> object.</p>
<p name="f4dd" id="f4dd" class="graf graf--p graf-after--p">I actually ran <code class="markup--code markup--p-code">phone_form.methods-Object.methods</code> to remove any methods that were generic to all Ruby objects and got:</p>
<pre name="988b" id="988b" class="graf graf--pre graf-after--p"><code class="markup--code markup--pre-code">&gt;&gt; phone_form.methods — Object.methods</code></pre>
<pre name="82e8" id="82e8" class="graf graf--pre graf-after--pre"><code class="markup--code markup--pre-code">=&gt; [:select, :index, :options, :label, :options=, :object, :multipart?, :submit, :object=, :fields, :to_partial_path, :to_model, :multipart, :date_select, :object_name, :time_select, :datetime_select, :fields_for, :text_field, :password_field, :hidden_field, :file_field, :text_area, :check_box, :radio_button, :color_field, :search_field, :telephone_field, :phone_field, :date_field, :time_field, :datetime_field, :datetime_local_field, :month_field, :week_field, :url_field, :email_field, :number_field, :range_field, :field_helpers, :field_helpers=, :multipart=, :button, :emitted_hidden_id?, :field_helpers?, :object_name=, :collection_select, :grouped_collection_select, :time_zone_select, :collection_radio_buttons, :collection_check_boxes, :model_name_from_record_or_class, :convert_to_model]</code></pre>
<p name="e8cd" id="e8cd" class="graf graf--p graf-after--pre">Playing around with some of the more promising sounding method names, I found the the <code class="markup--code markup--p-code">:index</code> method did indeed give me the index of the current form, and <code class="markup--code markup--p-code">:object</code> gave me the object that was tied to the form, so my final radio tag ended up looking like this:</p>
<pre name="f6ec" id="f6ec" class="graf graf--pre graf-after--p"><code class="markup--code markup--pre-code">&lt;%= radio_button_tag “contact[default_telephone]”, phone_form.index, phone_form.object.default %&gt;</code></pre>
<p name="da66" id="da66" class="graf graf--p graf-after--pre">The first argument (<code class="markup--code markup--p-code">“contact[default_telephone]”</code>) is the <code class="markup--code markup--p-code">name</code> attribute, which is the same for all phone number forms and ensures my radio buttons would act like radio buttons.</p>
<p name="fa3c" id="fa3c" class="graf graf--p graf-after--p">The second argument (<code class="markup--code markup--p-code">phone_form.index</code>) is the value passed in by the radio button, and is equal to the index of the form currently being worked on, and will be used by the controller to identify which phone number is the default one (as I will show soon).</p>
<p name="1b04" id="1b04" class="graf graf--p graf-after--p">The third argument (<code class="markup--code markup--p-code">phone_form.object.default</code>) determines whether the radio button should be selected by default. <code class="markup--code markup--p-code">phone_form.object</code> is the current <code class="markup--code markup--p-code">PhoneNumber</code> object the form is rendering, and the <code class="markup--code markup--p-code">.default</code> is the current value of whether this instance of <code class="markup--code markup--p-code">PhoneNumber</code> is default or not.</p>
<p name="d541" id="d541" class="graf graf--p graf-after--p">Now I just had to update my controller to let it use the information passed in by my form. I added the following two lines of code to the beginning of my <code class="markup--code markup--p-code">create</code> and <code class="markup--code markup--p-code">update</code> controllers:</p>
<pre name="e159" id="e159" class="graf graf--pre graf-after--p">def create<br> default_index = params[:contact][:default_telephone]<br> params[:contact][:telephones_attributes][default_index][:default] = true<br> # Create the new object….<br>end</pre>
